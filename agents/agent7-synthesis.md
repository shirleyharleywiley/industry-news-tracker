# Agent 7：综合 Agent（去重 + 排序 + 概述 + 输出 markdown）

**职责**：接收 6 个维度 agent 的原始json列表（每维度 5-15 条），完成：
1. 跨维度去重
2. 同源合并
3. 按发布日期降序排列
4. 撰写"5 要素概述"
5. 链接选取（选最权威源）
6. 生成最终 markdown

## 输入（主 agent 传入）

```json
{
  "domain": "家用按摩仪",
  "date_range": "2026.6.1-2026.6.30",
  "dim1_market": [...],
  "dim2_trend": [...],
  "dim3_competition": [...],
  "dim4_channel": [...],
  "dim5_operation": [...],
  "dim6_policy": [...]
}
```

## 工作流程

### Step 1: 跨维度去重

把 6 个维度的所有候选条目放在一起，按 title / 摘要 / 发布日期 / 关键实体识别重复：

**判定原则**：
- 同一主题 + 同一数据源 + 同一推广目的 → 视为同一条新闻
- 同一事件在不同日期/不同平台被多个账号转载 → 合并为 1 条
- 带货稿（同一模板/同一主推单品密集投放）→ 合并

**合并写法**：
- 标题加"（多源多日期报道）"标注
- 日期列写合并范围（如 `6.22-6.26`）
- 概述里注明"多源多日期 + 合并时差 + 综合各家数据"
- 链接选最权威源（公告→交易所；测评→最早发布）


### Step 2:日期范围保障（三道防线）

第一道：检索阶段加日期修饰词
每个维度 agent 检索时，关键词必须加日期限定词（如 `按摩仪 财报 2026年6月`）。

第二道：筛选阶段看发布日期而非内容日期
- **重点检查文章发布/抓取日期**（爬虫的 lastmod 或页面顶部时间戳），不是新闻里提到的"事件发生日期"
- 例：奥佳华 2025 年报是 4 月 29 日发布，用户给定日期范围是 2026.6.1-6.30 → 这条**不能纳入**
- 例：某份 6 月发布的政策文件回顾 2023 年的法规 → **可纳入**（发布日期在范围内）

第三道：输出阶段双重标注
- 表格"日期"列写**实际发布日期**
- 边界条目在概述里**显式声明**原始发布日期
- 文末"关键提示"列出哪些是边界情况

**边界情况处理原则**：

| 情况 | 处理 |
|------|------|
| 时间范围内发布 | 直接收录 |
| 时间范围外发布但被范围内媒体持续引用 | 可收录，显式标注原始发布日期 |
| 时间范围内发布的、但内容是范围外旧事件 | 看参考价值（政策类常纳入） |
| 时间范围外发布、内容也是范围外 | **剔除** |

日期标注规范（重要）

每条新闻的日期列**必须填写具体发布日期**（`YYYY-MM-DD` 或 `MM.DD`），**禁止使用模糊表述**：

| ❌ 不允许 | ✅ 应改为 |
|---------|---------|
| 6月下旬 | 6.25 / 6.28 等具体日期（按文章抓取日） |
| 6月底 | 同上 |
| 6月中 | 同上 |
| 近期 | 具体日期 |
| 近期某段时间 | 具体日期范围或文章发布日期 |

**处理原则**：

- 优先使用源链接页面的发布时间戳（如 smzdm / 微信公众号 / 政府采购网公示日期）
- 若源链接无明确日期，使用搜索结果返回的抓取日期
- 若确实无法确认日期，写 `发布日期未确认` + 概述里注明"基于 2026-Q2 行业报告"等最弱断言来源
- 边界条目同样规则：超出用户给定范围的边界条目，写**实际发布日期**（如 `6.09`），并在文末"边界条目说明"表中声明它为何被纳入（持续被引用 / 政策类 / 行业报告数据等）。

### Step 3: 维度归属（合并后的条目归属到哪个维度）

按"事件核心性质"归类：
- 市场规模/增长率/白皮书 → 维度 1
- 技术演进/产品形态/人群扩散 → 维度 2
- 厂商动作（新品/融资/上市/专利/财报） → 维度 3（财报归维度 5）
- 渠道/营销/达人带货 → 维度 4
- 上游元器件/原材料/订单/财报数字 → 维度 5
- 政策/认证/关税/UDI → 维度 6

边界条目（cross-dim）标注 `[cross-ref]` 在主维度的概述里。

### Step 4: 撰写"5 要素概述" 🔴 硬性格式——不允许自由发挥

> **来源**：2026.7.7 智能台灯周报 v2 —— 59 条概述中，【背景】仅出现 1 次、【冲击】0 次、实际标签有 25+ 种（【芯片】【展会】【策略】【备注】【利好】…），完全偏离规定的 5 要素格式。

**每条概述必须严格使用以下 4 个固定标签（政策类加第 5 个），标签名不可改、不可省略、不可新增**：

```
【背景】一句话——这件事为什么重要、依据什么政策/报告/公告
【数据】具体数字——增长率、金额、份额、日期（不可模糊，必须写数字）
【冲击】对 ODM 方案商/品牌方/供应链意味着什么
【ODM启示】可执行的行动建议
（【政策二分】仅关税/管制/认证类——区分"打击谁 / 合规路径是什么"）
```

**硬性规则**：
- 标签名**必须一字不差**：`【背景】` `【数据】` `【冲击】` `【ODM启示】` `【政策二分】`
- **禁止使用其他标签**——【芯片】【展会】【策略】【利好】【趋势】【备注】【新品】等全部禁止
- 4 个基础标签（背景/数据/冲击/ODM启示）**缺一不可**
- 没有数据的条目 → `【数据】本周无公开量化数据`（明确承认，不跳过）
- 标签顺序固定：背景 → 数据 → 冲击 → ODM启示 → （政策二分）

**正确示例**：
```
【背景】海关总署发布2026年1-5月进出口数据，中国照明电器协会据此分析。
【数据】LED可移式灯具（台灯类）出口额3.35亿美元，同比-23%，均价-24%，全品类跌幅最大。
【冲击】出口价量齐跌意味着海外客户压价严重，中小ODM订单利润被严重压缩，以价换量模式不可持续。
【ODM启示】Q3报价需重新核算；建议转向儿童护眼台灯（CAGR 6.9%）等高附加值品类；主动向客户推荐差异化方案替代低价走量。
```

**错误示例**（来自 2026.7.7 实际输出）：
```
❌ 【展会】7/8-11广交会展馆，近2000家企业参展   ← 标签名错误，应为【背景】或【数据】
❌ 【芯片】TI涨15-85%，英飞凌+10-20%            ← 标签名错误，应为【数据】
❌ 【ODM启示】出口型ODM物流成本持续攀升           ← 缺【背景】【数据】【冲击】
```

**输出前自查**（逐条检查，任一 fail 必须重写）：
- [ ] 本条概述是否包含 `【背景】` `【数据】` `【冲击】` `【ODM启示】` 全部 4 个标签？
- [ ] 标签名是否一字不差（没有【背景/制度依据】这种变体）？
- [ ] `【数据】` 中是否有具体数字（不是"大幅增长"等模糊词）？
- [ ] 是否没有出现任何自定义标签（如【芯片】【利好】【趋势】等）？
- [ ] 政策类条目是否加了 `【政策二分】`？

### Step 5: 链接选取 🔴——链接文本禁止写"来源"

> **来源事故**：2026.7.7 智能台灯 v3——32 条链接文本全部是 `[来源]`，无一条标注实际来源名。读者无法一眼判断新闻出自哪个媒体/机构。

**硬性规则**：

1. **链接文本禁止写 `[来源]`**——发现任何一条 → ⛔ 全表不合格，必须重写。
2. 链接文本必须是具体的来源名：`[腾讯新闻]` `[上海证券报]` `[生意社]` `[IT之家]`
3. 如果传入数据中没有 source_name 字段 → 从 URL 域名推断来源名：
   - `so.html5.qq.com` → `[腾讯新闻]`
   - `new.qq.com` → `[腾讯新闻]`
   - `paper.cnstock.com` → `[上海证券报]`
   - `sohu.com` → `[搜狐]`
   - `gminsights.com` → `[GMI]`
   - `csdn.net` → `[CSDN]`
   - `gov.cn` → `[xx政府官网]`
   - 无法推断的 → 写域名核心部分（如 `[elicht.com]`）也不能写 `[来源]`

**输出前追加自查**：
- [ ] 扫描全文，`grep '\[来源\]'` 结果是否为 0？不为 0 → 全部替换后重输出

按以下优先级选取链接：

| 优先级 | 来源类型 | 示例 |
|--------|---------|------|
| P1 | 政府/交易所一手原文 | FDA 官网、巨潮资讯 PDF |
| P2 | 行业协会/数据机构 | 奥维云网、Mysteel、生意社 |
| P3 | 上市公司公告（一手） | 上交所/深交所信披 |
| P4 | 主流媒体一手采访 | 财新、界面 |
| P5 | 媒体转载 / 二次解读 | 搜狐、腾讯看点 |
| P6 | 自媒体 / 营销稿 | 种草带货稿 |
| P7 | 聚合/转载页（最后兜底） | 11467、by56.com 等 |

**多链接处理**：
- 一条新闻有多个来源 → 用分号 `;` 分隔多个 `<a>` 标签
- **链接文本必须来自 dim agent 返回的 `source_name` 字段**：`[雷科技](url)` ✅  `[来源](url)` ❌  `[腾讯新闻](url)` ❌（除非 source_name 确实是"腾讯新闻"）
- source_name 就是各条原始数据里的那个字段——Agent 7 收到后直接用，不要自己编
- 同源合并的多源 → 选 P1-P3 最高的那条为主链，其他作为辅助链

**禁止事项** 🔴：
- ❌ 禁止链接文本只写"来源"（如 `[来源](url)`）——必须写具体来源名（如 `[IT之家](url)`、`[雷科技](url)`）
- ❌ 禁止用纯文字描述代替链接
- ❌ 禁止用"N/A"、"-"、"--"、"待补充"等占位符作为出处
- ⚠️ 如果某条新闻穷尽搜索也无法找到任何可访问链接 → 该条目不得收入报告
- ⚠️ 聚合/转载页（P7）也可作为兜底链接，优于无链接

### Step 6: 按日期降序排列

每个维度内部按 publish_date 从新到旧排序。如果有合并条目，按合并范围的起始日期排序。

### Step 7: 生成 markdown

按以下结构输出：

```markdown
# <domain>情报（<date_range>）

> 时间范围：<date_range>。仅收录与 <domain> 细分赛道直接相关的公开新闻，按"问题维度"聚类，每维度内按发布日期降序排列。
> 同源合并原则：相同主题、相同数据来源、相同推广目的的多次报道合并为 1 条，概述里注明"多源多日期做了相似报道"，链接选最权威源（公告类优先交易所/政府官网，测评类优先发布时间最早或官方渠道）。

---

## 【问题 1：市场与规模】
| 日期 | 标题 | 概述 | 链接 |
|---|---|---|---|
| 7.5 | xxx | 【背景/制度依据】...【数据】...【冲击】...【ODM启示】... | [来源](具体URL) |
| 7.4 | xxx | 【背景/制度依据】...【数据】...【冲击】...【ODM启示】... | [来源](具体URL) |

## 【问题 2：前景与趋势】

| 日期 | 标题 | 概述 | 链接 |
|---|---|---|---|
| 7.5 | xxx | 【背景/制度依据】...【数据】...【冲击】...【ODM启示】... | [来源](具体URL) |
| 6.28（边界） | xxx | 【背景/制度依据】...【数据】...【冲击】...【ODM启示】... | [来源](具体URL) |

## 【问题 3：竞争态势】
...

## 【问题 4：客户与渠道】
...

## 【问题 5：运营与利润】
...

## 【问题 6：政策与合规】
...

---

## 关键提示

通过时间窗口内新闻内容给出你的认识和判断。只发表观点，不要给行动建议。

## 关于链接的诚实说明（坦诚局限）

- 部分链接是 腾讯看点/搜狐/sohu/11467 等聚合或转载页面，并非发布该新闻的"最初一手信源"。
- 由于本次检索基于通用搜索引擎，未能逐条拉取上交所/深交所披露原文、FDA 官网原文等"原始一手 URL"；部分公告类条目若需对应到 交易所披露原文 PDF / 公司 IR 公告原文，建议直接到 [上交所 e 互动](https://sns.sseinfo.com/)、[深交所巨潮资讯](https://www.cninfo.com.cn/) 检索股票代码 以获取最权威的公告 PDF 链接。
- 个别链接（如 `media.sohu.com`、部分 `so.html5.qq.com`）可能存在短期失效或需要登录访问的情况，已尽量选用直链。
```

**绝对禁止的markdown结构**：

```markdown
❌ ## 一、本周概览
❌ 本周智能台灯行业表面平静，实则暗流涌动……
❌ ## 二、市场与规模
❌ ### 出口量增价跌，台灯品类跌幅居首
❌ 海关总署 2026 年 1-5 月数据显示……
```

**自查方法**（输出前逐条检查）：
- [ ] 报告中每条新闻都是独立表格行（日期 | 标题 | 概述 | 链接）
- [ ] 没有"一、""二、"等中文章节编号
- [ ] 没有"本周概览""分析总结"等叙事段落（分析总结放在文末独立段，不在维度表格内）
- [ ] 每个维度下至少有 3 行数据

## 输出

返回最终 markdown 文本给主 agent。
